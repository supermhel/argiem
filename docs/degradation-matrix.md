# Degradation matrix (M1 — Correctness gates)

*Every dependency FENGARDE has, what happens when it's down, and where that behavior is
implemented. Written from the actual code paths below, not aspirational — a row with no
implementation reference is a gap, not a claim. Part of the M1 correctness-gates milestone
(see SSOT.md §1).*

| Dependency | Down / unreachable | Recovery | Where |
|---|---|---|---|
| **Redis (bus)** | Any producer/consumer call raises; a worker's `/health` flips to `503 degraded` (was a static `ok`) so an orchestrator can restart it. No local buffering beyond the syslog ingest-edge spool (see below) — a wedged bus stalls the whole pipeline, by design (never silently drop). | Reconnect is the responsibility of the Redis client library / container restart; no custom backoff loop exists yet. | `services/shared/runner.py` (HealthState flip on consume/claim exceptions), `services/shared/bus.py` |
| **OpenSearch (write path)** | `OpenSearchStore.index()` retries a transient failure (connection error / 5xx) with bounded backoff, absorbing a brief outage inside one delivery instead of bleeding the event into the DLQ. A permanent 4xx (bad mapping/document) is **not** retried — surfaced immediately, since retrying would just burn redeliveries on an unfixable error. | Once OpenSearch answers again, the next bounded-retry attempt succeeds and indexing resumes; no replay step needed (delivery is at-least-once from the bus). | `services/ws3-indexer/storage/opensearch.py:24-70` |
| **OpenSearch (triage OCC)** | `find_alert_versioned` / `index_cas` use `_seq_no`/`_primary_term` optimistic concurrency; a stale write gets `409`, the retry loop re-reads and re-applies, bounded at `_CAS_MAX_RETRIES=5`. Exhaustion surfaces as an honest `409` to the client — never a silent drop. | Automatic within the bounded retry; a client-visible 409 requires the caller to re-request. | `services/ws3-indexer/storage/opensearch.py:154+`, unit-tested via fake transport only (`test_storage_cas.py`) — **not yet exercised against a live cluster under real concurrency**, see M1's chaos gate. |
| **Ollama (LLM triage)** | `make_llm()` probes `OLLAMA_URL` at startup: unreachable or unset -> falls back to `StubLLM` (deterministic offline verdict), never blocks the alert path. A reachable-at-startup-but-later-down Ollama is wrapped in `FallbackLLM(OllamaLLM, StubLLM)` so a runtime failure degrades per-call, not just at boot. | Once Ollama answers again, `FallbackLLM` resumes using it (no explicit reconnect logic needed — it's a per-call try). | `services/ws5-ai/llm_adapter.py:19,199-207` |
| **`fengarde-sec` HTTP report backend** | `REPORT_BACKEND=http` calls the external backend; a non-conforming or failed response is rejected by the structural validator (draft/disclaimer required) and WS-3 falls back to the builtin generic template — fail-open on the backend, fail-closed on the contract shape. | No retry needed — every request re-attempts the external call fresh; falls back per-request, not per-outage-window. | `contracts/reporting.md`, `services/ws3-indexer/reporting.py` |
| **Syslog UDP ingest under flood** | Token-bucket sheds excess datagrams (`SYSLOG_MAX_EVENTS_PER_SEC`, default 2000/s) *before* they reach the bus, rather than growing the Redis stream toward OOM. Shed-warning log itself throttled to 1/sec (can't become a logging DoS). | Opt-in zero-loss fallback: `BoundedSpool` (disk-backed JSONL, `SYSLOG_SPOOL_PATH`) captures shed/produce-failed events; a background thread replays them once capacity/connectivity returns. Once the spool itself fills, loss is distinctly counted (`events_lost`), never silently merged into the plain shed counter. Disabled by default. | `services/ws1-collectors/collectors/syslog_udp_server.py`, `services/ws1-collectors/collectors/spool.py` |
| **A5 enrichment (local IOC/geo files)** | Missing/malformed data file, bad IP, or any lookup error leaves the event **untouched and flowing** — additive-only, never blocks or corrupts normalization. | N/A — no state to recover, it's a stateless per-event lookup. | `services/ws2-normalization/enrichment/__init__.py:138`, `services/ws2-normalization/main.py:27` |
| **Allowlist files (`not_in` suppression)** | A missing or malformed allowlist file fails the rule **open** (keeps firing) but the suppression itself fails **closed** (never silently suppresses) — the asymmetry is deliberate: a broken suppression list must never disable detection. | Fix the file; next rule evaluation picks it up (no caching to invalidate). | `services/ws4-detection/engine.py:128-130,370` |
| **Malformed rule input (poison events)** | The whole condition evaluator fails closed to "no match" rather than raising and wedging the worker on redelivery — covers non-numeric/NaN/inf event time, far-future timestamps (window poisoning), unknown operators, malformed operator dicts. | N/A — a poison event simply never matches; no recovery step, no redelivery loop. | `services/ws4-detection/engine.py:77,207,257,387,402` (P0 hardening, `9e2745b`) |
| **A parser that raises on a bad record** | `normalize_one` wraps `parse()` so a raising parser dead-letters *one* record instead of aborting the whole batch; genuinely-ambiguous payloads are dead-lettered rather than silently mis-routed to the wrong parser. | Operator inspects the DLQ with `tools/dlq_peek.py` and can `--requeue` after fixing the root cause. | `services/ws2-normalization/parsers/__init__.py` (P0 hardening, `9e2745b`), `tools/dlq_peek.py` |

## Known gaps this matrix surfaces (not yet closed)

- **No documented Redis reconnect/backoff strategy** for the bus itself — every other row has one lever (retry, fallback, spool); Redis down currently means "the worker looks unhealthy until Redis or the worker container is restarted." Worth a jittered-reconnect follow-up if `make chaos` (below) shows it matters in practice.
- **OCC/CAS path** is unit-tested against a fake transport only — real concurrent-replica behavior under an actual OpenSearch outage is unverified. This is exactly what `make chaos` needs to exercise.
- This matrix is current as of commit `14d00ff`; keep it in sync with `SSOT.md` §2 (proven vs. claim) as gates close.
